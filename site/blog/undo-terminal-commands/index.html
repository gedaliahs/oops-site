How to undo rm, git reset, and other destructive terminal commands

March 2026

Every developer has that moment. You type rm -rf and hit enter a fraction of a second before your brain catches up. Or you run git reset --hard and realize your uncommitted work is gone. Or you mv a file over another one you needed.

The terminal has no undo. Every other interface — your text editor, your browser, your phone — has Ctrl+Z. The terminal, where you can do the most damage, has nothing.

Here's how to deal with it.

Can you recover a deleted file from the terminal?

It depends on what happened:

  • rm — the file is gone. Unlike dragging to Trash on macOS or Windows, rm bypasses the trash entirely. The data may still be on disk but recovering it requires forensic tools like testdisk or photorec, and even then it's unreliable — especially on SSDs.
  • git reset --hard — if the commits were pushed, they're still on the remote. If not, you can sometimes recover with git reflog and git cherry-pick. Uncommitted changes are usually gone.
  • mv overwrite — the overwritten file is gone. No recovery.
  • sed -i — the original file content is gone. If it was in git, you can git checkout the old version.

The common thread: recovery after the fact is unreliable at best. The real solution is to prevent the data loss in the first place.

The prevention approaches

Aliases like rm -i — makes rm ask for confirmation. Gets annoying fast, and most people alias it back or just hit 'y' without reading. Also only covers rm.

trash-cli / rip — replaces rm with a command that moves files to trash instead of deleting. Works, but only handles rm. You have to remember to use trash instead of rm, and it doesn't help with mv, sed, or git.

Git — tracks your source code, so you can recover committed files. But it doesn't help with uncommitted changes, untracked files, or anything outside a repo.

Time Machine / backups — work, but they're periodic. If you deleted a file 30 seconds ago and your last backup was an hour ago, you're out of luck.

The shell hook approach

What if the terminal itself could detect destructive commands and back up the affected files before they run?

That's what oops does. It installs a shell hook — a function that runs before every command you type. The hook pattern-matches the command:

  • Safe commands (ls, cat, git log, etc.) pass through with zero overhead. No subprocess is spawned.
  • Destructive commands (rm, mv, sed -i, git reset --hard, etc.) trigger a backup of the affected files before the command runs.

The backup uses hard links — a filesystem feature that creates a second reference to the same data on disk. This makes it instant regardless of file size (even gigabytes), and it uses no extra disk space until the original is deleted.

If you make a mistake, type oops to restore:

$ rm -rf src/ $ oops ✓ Undid: rm -r ~/project/src ↩ restored ~/project/src

What about AI coding agents?

This is increasingly relevant. Tools like Claude Code, Cursor, Aider, and Codex run shell commands on your behalf — including rm, git reset --hard, and mv. They sometimes get it wrong.

Since these tools run commands through your terminal's shell, oops catches them the same way it catches your own commands. If an agent deletes something it shouldn't have, type oops and it comes back.

Getting started

One command to install:

curl -fsSL oops-cli.com/install.sh | bash

The installer handles everything — downloads the binary, adds the shell hook, and creates the backup directory. Works on macOS and Linux with zsh, bash, or fish.

After installing, run oops tutorial for an interactive walkthrough, or just use your terminal normally and type oops when you need to undo something.
