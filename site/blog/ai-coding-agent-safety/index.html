How to protect your files from AI coding agents

March 2026

If you're using Claude Code, Cursor, Aider, or Codex, you've probably given an AI agent the ability to run shell commands on your machine. That includes rm, git reset --hard, mv, and everything else.

Most of the time it works great. But AI agents make mistakes. They misinterpret your intent, hallucinate file paths, or take destructive shortcuts. And unlike a human developer, they don't hesitate before running rm -rf.

What can actually go wrong

Here are real scenarios that happen with AI coding agents:

  • Deleting the wrong files. You ask the agent to clean up test files and it rm -rf's your src directory instead. The agent saw "clean up" and was too aggressive.
  • Resetting git history. The agent runs git reset --hard HEAD~5 to fix a merge conflict, wiping your last five uncommitted changes in the process.
  • Overwriting files with mv. The agent renames a file but the destination already exists. Your original file is gone.
  • In-place edits gone wrong. The agent runs sed -i on the wrong file, or with a bad pattern, corrupting your config.
  • Cleaning git state. git checkout . or git clean -fd to "start fresh" — wiping all your uncommitted work and untracked files.

The agent usually doesn't realize it made a mistake. It reports success and moves on. You discover the damage later.

Why existing protections don't help

Git only protects committed, tracked files. Uncommitted changes, untracked files, and anything outside a repo are unprotected.

Agent confirmation prompts help for some operations, but they add friction and slow down the workflow. And many people run agents in autonomous mode specifically to avoid prompts.

Backups / Time Machine are periodic. If the agent deleted something 30 seconds ago and your last backup was an hour ago, you're out of luck.

trash-cli only catches rm. It doesn't help with git reset --hard, mv overwrites, sed -i, or any of the other destructive commands agents use.

The shell hook approach

Every command an AI agent runs goes through your terminal's shell. If you have a shell hook that intercepts destructive commands, it catches agent commands the same way it catches yours.

That's what oops does. A preexec hook watches every command. When it sees something destructive, it backs up the affected files using hard links (instant, no extra disk space) before the command runs. If an agent deletes something it shouldn't have:

# Agent ran a destructive command ~ $ git reset --hard HEAD~3 ▲ git reset --hard # You notice the damage ~ $ oops ✓ Undid: git reset --hard Applied stash: stash@{0}

The agent doesn't know oops exists. It doesn't need to. oops works at the shell level, below any agent's awareness.

What it catches from agents

  • rm / rm -rf — backs up files before deletion
  • mv a b — backs up the overwrite target
  • sed -i — copies file before in-place edit
  • git reset --hard — creates a stash before reset
  • git checkout . — creates a stash
  • git clean -fd — stashes untracked files
  • git branch -D — logs the SHA before deletion
  • chmod / chown — records original permissions
  • > file redirects — snapshots before overwrite

Non-destructive commands (ls, cat, git log, npm install, etc.) pass through with zero overhead. No subprocess is spawned.

Install

curl -fsSL oops-cli.com/install.sh | bash

One command. Works on macOS and Linux with zsh, bash, or fish. The installer handles the binary, shell hook, and backup directory. Open a new terminal tab to activate.

Then just use your AI agent normally. oops runs silently in the background. When something goes wrong, type oops.
